# FaultNav Robotics Engineering Rules

Apply these rules to kinematics, estimation, sensor, ROS 2, simulator, and future hardware work.

## Units and conventions

Document and verify:

- metres versus millimetres;
- seconds versus milliseconds or nanoseconds;
- radians versus degrees;
- linear and angular velocity units;
- acceleration, force, torque, voltage, and current units where introduced;
- axis order, handedness, and sign conventions.

Do not infer units silently. Include units in variable names, schemas, configuration descriptions, or documentation where practical.

## Coordinate frames

For every transform or pose, establish:

- source frame;
- target frame;
- transform direction;
- timestamp;
- rotation representation;
- handedness;
- static versus dynamic ownership.

The current ROS 2 odometry node owns `odom -> base_link`. Future localisation components must not publish the same transform concurrently without an explicit architecture change.

Do not combine data from different frames without a verified transform.

## Time and synchronization

Handle and test where relevant:

- missing, duplicated, unsorted, delayed, and out-of-order timestamps;
- stale commands and stale measurements;
- clock drift and different sensor rates;
- interpolation and extrapolation assumptions;
- integration-step boundaries;
- simulation time versus wall time.

Time-window faults currently use half-open intervals evaluated at integration-step midpoints. Preserve or explicitly revise that convention.

## Numerical safety

Check:

- zero and near-zero denominators;
- angle wrapping;
- NaN and infinity;
- invalid dimensions;
- singular and ill-conditioned matrices;
- overflow and underflow;
- invalid normalization;
- covariance symmetry and positive semi-definiteness;
- integration stability;
- appropriate floating-point tolerances.

Important mathematical changes require analytical cases or an independent trusted implementation.

## Ground truth, measurements, and estimates

Maintain explicit separation:

- ground truth: reference trajectory generated by the documented model;
- measurements: quantised, noisy, biased, dropped, or faulted sensor outputs;
- estimates: odometry or filter results computed only from allowed measurements and models.

Fault injection must not silently alter the ground-truth reference used to calculate error.

## Sensor robustness

Test relevant combinations of:

- dropout;
- noise;
- spikes and outliers;
- saturation;
- bias and calibration error;
- invalid ranges;
- corrupted or incomplete messages;
- startup and shutdown transients;
- repeatability with fixed seeds.

Do not report a simulated fault model as a validated physical sensor model.

## Estimation requirements

For future EKF and fault-monitoring work:

- define the state vector and units;
- define process and measurement equations;
- document Jacobians or numerical differentiation;
- define covariance ordering and units;
- validate covariance symmetry and positive semi-definiteness;
- test zero-noise and known-bias cases;
- test innovation and Normalized Innovation Squared calculations independently;
- document gating thresholds and statistical assumptions;
- distinguish measurement rejection from fault diagnosis;
- compare against raw odometry and a non-fault-aware baseline;
- avoid tuning only on the final evaluation scenario.

## Control and hardware safety

- Enforce command and actuator limits when they are introduced.
- Preserve emergency-stop, timeout, watchdog, and fail-safe paths.
- Define behavior for stale, missing, contradictory, or implausible measurements.
- Keep simulation and hardware configuration separate.
- Require explicit approval before sending commands to physical hardware.
- Never weaken limits, interlocks, or safety behavior to improve a demonstration.
- Simulation success is not evidence of real-world safety.

## Reproducibility and evaluation

Record where relevant:

- dataset or bag version;
- configuration;
- random seed;
- Python, ROS 2, and simulator versions;
- hardware and calibration;
- evaluation scripts;
- metric definitions and baselines;
- excluded runs and reasons.

Do not report only the best run. Do not change metrics after viewing results without documenting the reason.
